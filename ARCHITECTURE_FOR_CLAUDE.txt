1. EXECUTIVE SUMMARY
================================================================================

FarmAdvisor is a multilingual AI-powered farm advisory web application for Tamil Nadu farmers. It is built with Next.js App Router and provides:

- Farmer registration and login with JWT cookie authentication and +91 India mobile number handling.
- AWS SNS phone OTP verification during registration.
- Farmer profile management with land details, address, phone, land area, land photo, and preferred language.
- Dashboard with live farm-weather, land/map information, market prices, soil health, suitable crop suggestions, recent chats, and advisor access.
- AI chat assistant with RAG knowledge retrieval, farmer memory, weather, soil report, crop plans, and optional crop image diagnosis.
- Crop planning workflow that creates AI-generated crop schedules based on farmer profile, crop, land, weather forecast, soil report, and assessment answers.
- Saved crop plans with active/inactive status. Multiple plans can be active.
- Current plan widgets derived from the active plan stage, active date, crop, soil report, weather, water, land, fertilizer, manure, and pesticide needs.
- Soil report upload and extraction using OpenAI vision/file processing. Original PDF/image and extracted JSON are stored in S3, and normalized report data is stored in DynamoDB.
- District and soil based crop suitability ranking through /api/crop-options and lib/crop-suitability.ts.
- Mandi/market price widget using the data.gov.in Agmarknet API, not S3.
- Daily SMS Draft dashboard widget plus manual send using AWS SNS.
- Automatic 8:00 AM IST daily SMS scheduler for active crop plans.
- Government schemes carousel and schemes API.
- Voice input/output using OpenAI speech APIs.
- SMS notification route using AWS SNS.
- Multilingual UI and AI responses for English, Hindi, and Tamil using next-intl plus an AI translation API.

The system combines local UI translations, dynamic LLM translation, AWS storage, DynamoDB persistence, OpenAI models, Open-Meteo weather data, Qdrant vector search, and S3-based media storage.

Current important implementation notes:
- Weather uses farmer_profiles.land_coordinates only. The old Chennai fallback has been removed.
- Dashboard map displays the farmer's selected land polygon and supports zoom controls.
- Recent chats load into the existing dashboard chatbot via /dashboard?chat=<id>; /chat/[id] redirects there.
- Soil upload exists only on Profile. Dashboard and crop-plan pages display/translate saved soil report details.
- New-plan crop entry is a plain text input. The farmer can enter any crop name; the app uses exactly that input.
- Suitable crops are shown as a dashboard widget and are ranked by saved farm address/district plus soil pH and N/P/K.
- Dashboard Daily SMS Draft is shown horizontally below Suitable Crops and follows the current page language.
- Dashboard active crop plan widgets show all active plans, not only one.
- Dashboard active crop plan text has a local Tamil/Hindi fallback for crop names, stage names, tasks, and alert advice when API translation is unavailable.
- Daily SMS preview/manual send/automatic send include all active crop plans.
- Automatic Daily SMS uses the farmer's saved preferred_language.
- Farmer profile JSON, crop plan JSON, original soil report files, and soil extraction JSON are mirrored into farmer-specific S3 folders.
- Farmer ID format is TN followed by exactly 11 digits, e.g. TN12345678901.
- Profile farmer-memory facts are stored in English for stable LLM prompting, but the Profile page renders known land facts in the current page language.

Data update model:
- farmer_profiles is the source of truth for farmer identity, contact, preferred language, address, land polygon, land area/type, land image key, and memory facts.
- Profile registration/update writes to DynamoDB and mirrors safe profile JSON to S3 at <farmerId>/profile/profile.json.
- farmer memory is updated by app/api/farmer/memory/route.ts and by chat memory extraction in app/api/chat/route.ts. The same memory is injected into chat and crop-plan prompts.
- soil_reports is the source of truth for extracted lab soil values. Uploading a soil report stores the original file in S3, stores extraction JSON in S3, and stores normalized report data in DynamoDB.
- crop_plans is the source of truth for crop plan schedules and active/inactive state. Create/save/activate/deactivate mirrors plan JSON to S3; delete removes the mirrored S3 plan JSON.
- Weather and mandi prices are not stored as long-term farmer data. Weather is fetched live/near-live from Open-Meteo using the farm centroid. Mandi prices are fetched from data.gov.in Agmarknet API.
- Daily SMS preview reads current data only. Manual send and automatic 8:00 AM IST cron send through AWS SNS.

Example data-to-advice path:
- Farmer TN12345678901 registers with 2 acres near Kalasapakkam and Tamil as preferred language.
- The farmer uploads a soil report showing pH 6.6, EC 0.11, low nitrogen, low phosphorus, medium potassium, and low organic carbon.
- The uploaded report is saved as:
  - <farmerId>/soil-reports/<reportId>/original.<ext>
  - <farmerId>/soil-reports/<reportId>/extraction.json
  - DynamoDB soil_reports normalized fields.
- The farmer has two active crop plans: Chilli and Black gram.
- Dashboard Active Crop Plans renders both active plans, each with its current stage, stage dates, tasks, weather risk, and action guidance.
- Dashboard Soil Health reads the same uploaded soil report and explains pH/N/P/K status in the selected language.
- Suitable Crops ranks crops using district/address plus pH/N/P/K; pulses such as Black gram or Green gram can be suggested because they fit pH 6.6 and can help improve nitrogen.
- Daily SMS combines both active plans into one short message:
  - Chilli current stage and field work
  - Black gram current stage and field work
  - weather warning or safe-action point
  - soil pH/N/P/K guidance
  - fertilizer/manure/pest reminders
- If Black gram is deactivated or deleted, it is removed from DynamoDB active-plan queries, the S3 plan mirror is updated/removed, and the dashboard/SMS only use the remaining active plans.

4. ROUTING AND AUTHENTICATION ARCHITECTURE
================================================================================

Authentication model:
- Farmers register using name, phone, password, address, land boundary, typography, land area, and optional land image.
- Passwords are hashed with bcryptjs.
- Login validates phone/password and creates a signed JWT.
- JWT is stored in an httpOnly cookie named auth_token.
- Cookie defaults:
  - path: /
  - maxAge: 7 days
  - sameSite: lax
  - secure only when COOKIE_SECURE=true

JWT payload:
- farmerId
- phone
- name

Middleware:
- middleware.ts protects all non-public routes.
- Public paths include:
  - /
  - /login
  - /register
  - /api/auth
  - /api/schemes
  - /api/health
  - /_next
  - /favicon
- If auth_token is missing for a protected page/API path, user is redirected to /login?from=<current-path>.

Important auth API routes:
- POST /api/auth/register:
  - Validates farmer profile fields with Zod.
  - Requires farmerId to match TN followed by exactly 11 digits.
  - Checks duplicate farmer_id and phone.
  - Uses DynamoDB phone-index where available, with scan fallback.
  - Stores farmer profile in DynamoDB farmer_profiles.

- POST /api/auth/login:
  - Looks up farmer by phone.
  - Compares bcrypt password.
  - Sets auth_token cookie.
  - Returns safe farmer data.

- GET /api/farmer/profile:
  - Reads authenticated farmer profile.
  - Removes password_hash before response.

- PATCH /api/farmer/profile:
  - Updates editable profile fields:
    - name
    - typography
    - landAreaAcres
    - preferredLanguage
    - landPictureS3Key

5. INTERNATIONALIZATION AND LANGUAGE ARCHITECTURE
================================================================================

Supported locales:
- en: English
- hi: Hindi
- ta: Tamil

Static UI translation:
- next-intl loads messages from:
  - public/locales/en/common.json
  - public/locales/hi/common.json
  - public/locales/ta/common.json
- lib/i18n.ts provides request config.
- LanguageSwitcher changes the selected locale and stores it client-side/cookie-based for UI rendering.

Dynamic content translation:
- Dynamic LLM/database content is translated using POST /api/translate.
- Translate route supports kind:
  - crop_plan
  - soil_report
  - current_plan_details
- For crop plans, it translates farmer-facing strings inside generated plan objects.
- For soil reports, it translates plainLanguageSummary, keyFindings, and recommendations while preserving technical values such as pH, EC, NPK labels, numbers, units, and chemical names where needed.
- For current plan details, it translates farmer-facing widgets and arrays while preserving dates, numbers, units, pH, EC, NPK, and chemical symbols.
- There is a local rule-based fallback for common crop-plan/weather phrases if OpenAI translation fails or returns unchanged text.

Important behavior expectation:
- When the farmer changes language, both static UI labels and dynamic database/LLM content should be re-rendered or translated to the selected locale.
- Crop plan milestones, weather warnings, current plan widgets, soil report text, chat assistant content, government scheme text, and crop detail widgets should all use the selected language.
- Dashboard Daily SMS Draft refetches with GET /api/daily-sms?locale=<selected-locale>, so the preview follows the page language.
- Manual Daily SMS Send posts { locale } to /api/daily-sms, so the manually sent message follows the page language.
- Automatic scheduled SMS uses the saved farmer preferred_language, not the temporary page language.
- Dashboard Active Crop Plans card localizes active plan display text and applies local Tamil/Hindi fallbacks for common crop-plan phrases when API translation is incomplete.

S3 bucket:
- Bucket from S3_BUCKET_MEDIA, fallback farm-advisor-media.
- Region from AWS_REGION, fallback ap-south-1.
- Used for:
  - farmer land images
  - soil report PDFs/images
  - soil extraction JSON
  - safe farmer profile JSON mirror
  - crop plan JSON mirrors
  - crop images for diagnosis where applicable

S3 key pattern:
- buildS3Key(farmerId, type, filename)
- General shape: <farmerId>/<type>/<timestamp>-<filename>
- Farmer-scoped canonical JSON/file folders:
  - <farmerId>/profile/profile.json
  - <farmerId>/crop-plans/<planId>/plan.json
  - <farmerId>/soil-reports/<reportId>/original.<ext>
  - <farmerId>/soil-reports/<reportId>/extraction.json

S3 mirroring behavior:
- Registration and profile update mirror safe profile JSON to S3.
- Crop plan create/save/activate/deactivate mirrors plan JSON to S3.
- Crop plan delete removes the mirrored S3 plan JSON.
- Soil report upload stores the original file and extracted JSON in the report folder.

Duplicate soil report behavior:
- Soil upload computes SHA-256 hash of the uploaded file.
- If the same farmer uploads the same PDF/image again, the system does not upload a duplicate file and does not create duplicate JSON.
- Existing matching report is marked current and returned.

8. RAG / KNOWLEDGE BASE ARCHITECTURE
================================================================================

Files:
- lib/ai/rag.ts
- lib/ai/vectorstore.ts
- scripts/ingest-kb.ts

Vector database:
- Qdrant
- URL from QDRANT_URL, fallback http://localhost:6333
- Collection from QDRANT_COLLECTION, fallback farm_docs
- Vector size: 1536
- Distance: cosine

Retrieval flow:
1. User message is embedded using OpenAI embeddings.
2. Qdrant search returns top k similar chunks.
3. retrieveContext formats chunks as numbered source sections.
4. Advisor chat prompt includes retrieved context.
5. If Qdrant is unavailable, retrieval returns an empty string and chat continues without RAG.

Knowledge ingestion:
- scripts/ingest-kb.ts ingests knowledge documents/chunks.
- Chunks are embedded with OpenAI.
- Chunks are upserted into Qdrant with source metadata.

9. WEATHER ARCHITECTURE
================================================================================

Weather provider:
- Open-Meteo forecast API
- Reverse geocoding through Nominatim

File:
- lib/weather.ts

API:
- GET /api/weather

Weather route flow:

Weather data structure:

  - description

  - wind_speed
  - city/farm label
- Forecast day:
  - date
  - temp_min
  - temp_max
  - description
  - emoji
  - rain_mm

Crop plan weather integration:
- lib/crop-plan-weather.ts annotates milestones with weather information and warning text.
- Weather is used to avoid heavy rain, waterlogging, thunderstorm days, spraying/fertilizing during storm/rain, and unsuitable field operation windows.
- Dashboard weather widget shows a weekly forecast summary and highlights thunderstorm/rain risk in selected language.

10. SOIL REPORT ARCHITECTURE
================================================================================

Frontend:
- components/profile/SoilReportUpload
- Located in profile page.
- Farmer uploads PDF/image soil report.
- Supports selected language.
- Shows extracted result cards:
  - What this means
  - Key findings
  - pH
  - Nitrogen
  - Phosphorus
  - Potassium
  - AI recommendations

Backend:
- POST /api/soil/upload
- GET /api/soil

Upload flow:
1. Farmer selects PDF/image soil report.
2. Client sends multipart form data with file and locale.
3. API verifies auth_token.
4. API checks OPENAI_API_KEY.
5. File buffer is read.
6. SHA-256 hash is computed for duplicate detection.
7. Existing farmer soil reports are checked for same hash.
8. If duplicate:
   - Previous reports are marked not current.
   - Existing duplicate is marked current.
   - Existing soilData, reportId, s3Key, and jsonS3Key are returned.
   - PDF/image is not uploaded again.
   - JSON is not uploaded again.
9. If new:
   - Original report is uploaded to S3.
   - PDF extraction uses OpenAI file/document handling first.
   - If PDF extraction fails, pdf-parse text fallback can be used.
   - Image extraction uses OpenAI vision.
   - OpenAI returns strict JSON.
   - Extracted JSON is uploaded to S3.
   - Normalized fields are stored in DynamoDB.
   - Previous reports are marked not current.
   - New report is marked current.

Soil report fields used by crop planning:
- pH
- electrical conductivity / salinity
- organic carbon
- nitrogen
- phosphorus
- potassium
- micronutrients such as zinc, copper, iron, manganese
- plain-language summary
- key findings
- recommendations

How soil report influences crop plan:
- Land preparation guidance uses soil pH, EC, organic carbon, and drainage needs.
- Fertilizer recommendations use N/P/K status.
- Manure/organic matter recommendations use organic carbon status.
- Micronutrient recommendations use zinc/copper/iron/manganese findings.
- Irrigation/water advice can use salinity/EC and drainage conditions.
- Crop plan prompts explicitly say not to invent missing soil data.

11. CROP PLAN ARCHITECTURE
================================================================================

Frontend:
- app/crop-plan/page.tsx
- components/crop-plan/*

Main features:
- Create crop plan from user inputs.
- Choose farmer state:
  - planning_unsure
  - planning_specific
  - mid_grow
- Ask assessment questions.
- Generate AI crop plan.
- Save crop plan.
- View saved plans.
- Mark one or more plans active.
- Mark active plans inactive.
- Show selected plan details.
- Show current stage widget based on active_from date.
- Show crop/land/water/fertilizer/pesticide/manure requirement widgets above the plan assistant.
- Plan assistant can answer questions and modify plan.

Backend:
- GET /api/crop-plan
- POST /api/crop-plan
- DELETE /api/crop-plan?planId=<id>
- POST /api/crop-plan/chat

Plan generation flow:
1. User selects crop/planning state and fills required assessment.
2. API validates input with Zod.
3. API reads farmer profile from DynamoDB.
4. API reads latest/current soil report.
5. API reads durable farmer memory.
6. API fetches farm weather forecast from land centroid.
7. Prompt is built with:
   - crop name
   - farmer state
   - land area
   - address
   - typography
   - farmer assessment
   - soil report details
   - weather forecast
   - memory facts
   - selected locale/language
8. OpenAI is asked to return structured JSON.
9. Returned plan is normalized.
10. Milestones are annotated with weather warnings.
11. Plan is saved in DynamoDB crop_plans.
12. API returns normalized planData.

Expected crop plan shape:
- cropName
- startDate
- harvestDate
- sellWindow
- storageNotes
- budgetEstimate
- weatherAlerts
- milestones array

Milestone shape:
- id
- label
- date
- endDate
- durationDays
- tasks
- estimatedCost
- weatherRequirement
- weatherSummary
- alertAdvice

Active plan behavior:
- Saved plans can be activated.
- Multiple saved plans can be active.
- Active plan stores active_from.
- Current stage calculation must be based on active_from, not the real current date unless active_from is the current date.
- Farmer can deactivate an active plan, changing status back to planned.
- Current plan widgets are shown only when at least one plan is active.

Current stage/widget logic:
- Current stage is derived from active_from and milestone durations.
- If active_from changes or the plan is activated later, the stage timeline shifts from that activation date.
- The current stage changes as days pass relative to active_from.
- Widgets should use actual plan data and soil report data, not generic recommendations.

Current widgets:
1. Current stage card:
   - crop
   - stage name
   - starting date derived from active_from and milestone position
   - priority/risk
   - stage task summary

2. Crop info widget:
   - crop name
   - harvest date
   - sell window
   - budget
   - stage/crop context

3. Land/current requirement widget:
   - land condition based on milestone tasks
   - water/irrigation requirement
   - weather restrictions

9. Return assistant response.

   - chat messages

19. SMS / NOTIFICATION ARCHITECTURE
================================================================================

API:
- POST /api/notifications/sms
- GET /api/notifications/sms
- GET /api/daily-sms
- POST /api/daily-sms
- GET /api/cron/daily-sms

POST flow:
1. Authenticate farmer.
2. Validate SMS type and message.
3. Read farmer profile.
4. Use profile phone number.
5. Send SMS through AWS SNS.

SMS types:
- weather_alert
- milestone_reminder
- custom

GET flow:
- Admin-only placeholder weather check endpoint protected by x-admin-secret.
- Intended for cron/scheduler to check active crop plans and weather risk.

Daily SMS preview/send:
1. Dashboard calls GET /api/daily-sms?locale=<page-locale>.
2. API reads farmer profile, all active crop plans, latest soil report, and farm weather.
3. lib/daily-sms.ts drafts a short numbered SMS using all active crop stages, weather risk, soil pH/N/P/K, nutrient/fertilizer reminders, and pest/disease check.
4. The draft is translated/generated in the current page language.
5. Manual Send calls POST /api/daily-sms with { locale } and sends through AWS SNS to the farmer phone.

Automatic daily SMS:
1. vercel.json schedules /api/cron/daily-sms at 30 2 * * *.
2. 30 2 UTC equals 8:00 AM IST.
3. Cron route validates Vercel cron header or CRON_SECRET.
4. It scans eligible verified farmers, finds all active crop plans, drafts messages using each farmer's saved preferred_language, and sends through AWS SNS.

23. SECURITY AND OPERATIONAL NOTES
================================================================================

Authentication:
- Protected routes rely on auth_token cookie.
- Password hash is never returned from profile API.
- Admin seed and scheduler-style endpoints require ADMIN_SECRET.

Data isolation:
- Farmer-specific data is keyed by farmer_id.
- S3 keys are namespaced by farmer_id.
- API routes should always derive farmerId from JWT, not client input.

LLM safety:
- Prompts require not inventing values.
- Soil report values must be used when available.
- Missing soil report must be acknowledged.
- JSON mode is used for structured outputs.
- Translation route preserves technical values.

Reliability:
- RAG retrieval fails gracefully if Qdrant is unavailable.
- Crop plan generation has fallback plan builder if OpenAI fails.
- Soil upload has duplicate detection by file hash.
- Weather API responses are cached with Next.js revalidate settings.

Potential future improvements:
- Add scheduled job to scan active crop plans and weather, then send SNS alerts.
- Add stricter file size validation on soil upload API route.
- Add DynamoDB GSIs for current soil report and active crop plans if scale increases.
- Add persistent translated variants for crop plans and soil reports to reduce repeat translation calls.
- Add observability logs/metrics for OpenAI, S3, DynamoDB, and weather failures.
