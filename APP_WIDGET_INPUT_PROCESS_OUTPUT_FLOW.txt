LATEST IMPLEMENTATION CHANGES - 2026-06-06
================================================================================

Latest updates added on 2026-06-07:

Dashboard UI and layout:
- The dashboard now uses a wider max-width layout with cleaner spacing.
- Main farm widgets stay in the left/main content area.
- Chatbot and recent chats stay in the right dashboard rail on large screens.
- Daily SMS Draft was moved out of the chat rail and placed horizontally below Suitable Crops.
- Daily SMS Draft uses a responsive layout:
  - mobile: stacked draft + controls
  - desktop: draft text on the left, language/phone/send controls on the right
- Dashboard crop-plan widgets now use all active crop plans, not only the first active plan.
- If any active plan is made inactive or deleted, its dashboard details and SMS content are removed.

Daily SMS:
- app/api/daily-sms/route.ts supports:
  - GET /api/daily-sms?locale=<en|hi|ta> to preview the draft in the current page language.
  - POST /api/daily-sms with { locale } to manually send the current page-language SMS.
- lib/daily-sms.ts builds one SMS using all active crop plans, weather forecast, soil report, current stages, fertilizer/nutrient needs, and pest/disease reminder.
- Automatic daily SMS still uses the farmer's saved preferred_language from registration/profile.
- Manual dashboard preview/send follows the current page language selected in the app.
- app/api/cron/daily-sms/route.ts sends automatic SMS to farmers with verified phone numbers and active crop plans.
- vercel.json schedules /api/cron/daily-sms at 30 2 * * *, which is 8:00 AM IST.

Language behavior:
- Daily SMS Draft title, loading text, metadata, send status, send button, and generated SMS body follow the selected page language.
- Dashboard Active Crop Plans card translates crop names, stage names, task details, and alert text to the page language, with a local Tamil/Hindi fallback if API translation is unavailable.
- Market price crop names, known market names, known village/market places, and weather labels are localized where mappings or translation are available.
- Dynamic dashboard content should refetch/retranslate when the selected app language changes.

Farmer ID:
- Farmer ID must be TN followed by exactly 11 digits, for example TN12345678901.
- Register form uppercases and cleans the Farmer ID while typing.
- Register API enforces the same TN + 11 digit format before saving.

Farmer data S3 mirroring:
- Farmer registration and profile updates are mirrored to S3 as JSON.
- Crop plan create/save/activate/deactivate actions mirror the plan JSON to S3.
- Crop plan delete also deletes the mirrored S3 plan JSON.
- Soil report upload stores the original report file in S3 and stores extracted JSON beside it.
- S3 farmer folder structure:
  - <farmerId>/profile/profile.json
  - <farmerId>/crop-plans/<planId>/plan.json
  - <farmerId>/soil-reports/<reportId>/original.<ext>
  - <farmerId>/soil-reports/<reportId>/extraction.json

How saved data is updated:
- Farmer registration:
  - Input: Farmer ID, name, phone, OTP, address, land boundary, land area, land type, preferred language, password.
  - Update: app/api/auth/register/route.ts creates the farmer profile in DynamoDB farmer_profiles.
  - Mirror: safe profile JSON is written to <farmerId>/profile/profile.json in S3.
  - Output: login/dashboard/profile use the saved profile and language.
- Profile edit:
  - Input: edited profile fields.
  - Update: app/api/farmer/profile/route.ts updates DynamoDB farmer_profiles.
  - Mirror: <farmerId>/profile/profile.json is overwritten with the latest profile JSON.
  - Output: dashboard weather, land map, crop plan, daily SMS, and chat context use the new profile data.
- Farmer memory:
  - Input: durable facts extracted from chat, plus notes manually added on Profile.
  - Update: app/api/farmer/memory/route.ts updates the memory array inside farmer_profiles.
  - Output: Profile shows facts in the current page language; chat and crop-plan prompts receive the stored facts as farmer context.
- Soil report upload:
  - Input: PDF/JPEG/PNG lab soil report uploaded only on Profile.
  - Update: app/api/soil/upload/route.ts checks duplicate hash, uploads the original file to S3, extracts values with OpenAI, stores normalized data in DynamoDB soil_reports, and uploads extraction JSON to S3.
  - Mirror:
    - <farmerId>/soil-reports/<reportId>/original.<ext>
    - <farmerId>/soil-reports/<reportId>/extraction.json
  - Output: Profile, Dashboard Soil Health, Crop Plan land state, Suitable Crops, chatbot, crop-plan generation, and Daily SMS use the latest/current soil report.
- Crop plan create/save:
  - Input: crop name, assessment answers, farmer profile, weather, soil report, memory, and optional selected land/crop details.
  - Update: app/api/crop-plan/route.ts creates or updates the crop plan in DynamoDB crop_plans.
  - Mirror: <farmerId>/crop-plans/<planId>/plan.json is written to S3.
  - Output: Crop Plan page shows the plan; active plans flow to dashboard widgets and daily SMS.
- Crop plan activate/deactivate/delete:
  - Input: user action from the Crop Plan page.
  - Update: status changes are saved in DynamoDB crop_plans.
  - Mirror: activation/deactivation overwrites the S3 plan JSON; delete removes the S3 plan JSON.
  - Output: inactive/deleted plans disappear from dashboard Active Crop Plans, Today Action Banner, and Daily SMS.
- Weather:
  - Input: saved land polygon in farmer_profiles.land_coordinates.
  - Update: no app write is needed; dashboard/API fetches live Open-Meteo data using the farm polygon centroid.
  - Output: weather widget, crop-plan alerts, chat context, and SMS use the current farm weather.
- Mandi prices:
  - Input: selected/search crop name and state.
  - Update: no app write is needed; app/api/market-prices/route.ts fetches data.gov.in Agmarknet API on request.
  - Output: dashboard market widget shows translated crop/market/arrival text where mappings or translation are available.
- Daily SMS:
  - Input: farmer profile, verified phone, all active crop plans, latest soil report, live weather, and preferred/current locale.
  - Update: preview does not write data. Manual send posts to AWS SNS. Automatic cron sends at 8:00 AM IST.
  - Output: dashboard preview and actual SMS include all active plans and follow the selected/saved language.

Example: uploaded soil report + Chilli and Black gram active plans:
- Farmer:
  - farmer_id: TN12345678901
  - land: 2 acres near Kalasapakkam
  - preferred language: Tamil
  - verified phone: +91XXXXXXXXXX
- Uploaded soil report:
  - pH: 6.6
  - EC: 0.11
  - Nitrogen: Low
  - Phosphorus: Low
  - Potassium: Medium
  - Organic carbon: Low
- Active plan 1: Chilli
  - Current stage: Land Preparation
  - Dashboard Active Crop Plans shows Chilli stage, date range, land-preparation tasks, and any weather alert.
  - Soil Health uses pH/N/P/K from the uploaded report and explains that N and P need correction.
  - Daily SMS includes Chilli action points such as clearing weeds, deep ploughing, drainage, and avoiding open-field work during storm/heavy-rain warnings.
- Active plan 2: Black gram
  - Current stage: Land Preparation or the active milestone for that crop.
  - Dashboard Active Crop Plans also shows Black gram, not only Chilli.
  - Suitable Crops may recommend pulses like Black gram/Green gram because pulses can improve nitrogen and pH 6.6 fits many pulse crops.
  - Daily SMS includes Black gram points together with Chilli points in the same message.
- If Black gram is deactivated:
  - DynamoDB crop_plans status changes to inactive.
  - S3 <farmerId>/crop-plans/<planId>/plan.json is updated.
  - Dashboard and Daily SMS remove Black gram and keep only the remaining active plans.
- Language behavior:
  - Stored memory may remain English for LLM consistency.
  - Farmer-facing widgets translate rendered facts, crop names, stage names, soil text, alerts, SMS body, and buttons to the current app language.

Phone number handling:
- All farmer mobile number inputs visually show the India country code prefix +91.
- UI still stores and sends the local 10 digit number internally.
- lib/phone.ts normalizes +91XXXXXXXXXX, 91XXXXXXXXXX, 0XXXXXXXXXX, and XXXXXXXXXX into a 10 digit phone.
- SNS helpers in lib/aws/sns.ts convert phone numbers to E.164 format +91XXXXXXXXXX before SMS/OTP calls.

OTP / SNS verification:
- Register details page includes Send OTP / Resend OTP flow.
- Resend is disabled for 5 minutes after sending.
- Register API verifies the OTP before creating the farmer account.
- Standalone verify OTP page also uses +91-prefixed phone entry.

Location and weather:
- Weather uses the farmer's saved land boundary coordinates from farmer_profiles.land_coordinates.
- The centroid of the selected land polygon is used for current weather, forecast, crop plan weather annotation, crop-plan chat, and chatbot weather context.
- The hidden Chennai fallback was removed. If land coordinates are missing, weather is unavailable instead of using another location.
- Weather is live/near-live from Open-Meteo with periodic Next.js revalidation.
- Reverse geocoding labels the weather location as "Farm near <place>" using Nominatim, based on the farm coordinates.

Dashboard layout:
- Recent Chats was moved below the chatbot in the right dashboard column.
- Chatbot remains in-place; recent chat clicks load that conversation into the existing dashboard chatbot via /dashboard?chat=<id>.
- /chat/[id] redirects to the dashboard chat instead of showing a separate chat page.
- Dashboard land map shows the farmer-selected land polygon and supports map zoom controls and double-click zoom.

Soil report behavior:
- Soil report upload is only on the Profile page.
- Profile fetches /api/soil with no-store caching and shows saved soil report details when present.
- Dashboard Soil Health and Crop Plan land state widgets read the latest soil report and translate soil report summary/findings/recommendations based on the current app language.
- If a duplicate soil report is uploaded, the profile page shows that the details are already present.

Crop planning:
- In the New Plan flow, when a farmer chooses that they have a crop in mind, the crop field is now a plain input box, not a dropdown.
- The app uses exactly the crop name the farmer types.
- Enter key submits/advances where appropriate in login/register/search/crop plan inputs.
- Current ongoing stage progress shows only stage progress: completed green, current brand green, future gray. Risk is shown only in the Risk badge.

District/soil crop suitability:
- lib/crop-suitability.ts ranks crop options using farmer address/district and soil report values.
- /api/crop-options returns suitable crop names and reasons using address + pH + N/P/K status.
- Dashboard includes SuitableCropsWidget to show other crops the farmer can consider for this farm.

Market prices:
- Mandi prices are fetched from data.gov.in Agmarknet API in app/api/market-prices/route.ts.
- The app does not use S3 for mandi prices.
- DATA_GOV_IN_API_KEY is optional; if empty, the route falls back to a shared sample key.
- Commodity aliases are handled, e.g. Chilli -> Green Chilli and Rice -> Paddy(Dhan).

1. LOGIN PAGE
================================================================================

Page file:

Main widgets:
- Language switcher
- Phone input
- Password input with show/hide toggle
- Login button
- Error box

1.1 Language Switcher
Input:
- User chooses language: English, Hindi, or Tamil.
Source:
- components/layout/LanguageSwitcher.tsx
- Locale messages in public/locales/*/common.json
Processing:
- Updates app locale cookie/client state.
- Re-renders text through next-intl.
Output:
- Login page labels and placeholders change language.

1.2 Phone Input
Input:
- Farmer enters an India mobile number with the visible +91 prefix.
Source:
- User typing in login form.
Processing:
- UI accepts the local 10 digit number while showing +91.
- lib/phone.ts normalizes +91XXXXXXXXXX, 91XXXXXXXXXX, 0XXXXXXXXXX, and XXXXXXXXXX into a 10 digit local phone value.
- Sent to POST /api/auth/login.
Output:
- Phone becomes the lookup key for farmer account.

1.3 Password Input
Input:
- Farmer enters password.
Source:
- User typing in login form.
Processing:
- Stored in React state as password.
- Show/hide button only changes input type between password and text.
- Sent to POST /api/auth/login.
Output:
- Password is validated server-side against bcrypt hash.

1.4 Login Button
Input:
- phone
- password
Source:
- Login form state.
Processing:
- Calls app/api/auth/login/route.ts.
- API validates input with Zod.
- API looks up farmer by phone using app/api/auth/farmers.ts.
- DynamoDB query uses phone-index if available, with scan fallback.
- bcrypt compares password with password_hash.
- JWT is created by lib/auth.ts.
- auth_token cookie is set.
Output:
- Success: user goes to /dashboard.
- Failure: error message is displayed.

1.5 Error Box
Input:
- API error or network error.
Source:
- Login API response.
Processing:
- React state error is set.
Output:
- Farmer sees invalid credentials, failed login, or network error.

2.1 Step Indicator

Source:
- Register page React state.
Processing:
- stepIndex decides active/completed step.
Output:

2.2 FarmerDetailsForm
File:
- components/register/FarmerDetailsForm.tsx
Input:
- farmerId
- name
- phone
- phoneOtp
- address
- typography / land type
- landAreaAcres
- preferredLanguage
Source:
- Farmer form fields.
- Labels and validation messages from next-intl register namespace.
Processing:
- react-hook-form collects fields.
- Zod validates:
  - farmerId must match TN followed by exactly 11 digits, e.g. TN12345678901
  - name minimum length
  - phone must be a valid India mobile number with +91 display and 10 digit normalized storage
  - OTP must be verified before account creation
  - address minimum length
  - landAreaAcres positive number
- Send OTP calls SNS-backed OTP API. After one send, the button changes to Resend OTP and stays disabled for 5 minutes.
- preferredLanguage is saved as preferred_language on the farmer profile and is used by the automatic 8:00 AM SMS job.
- On success, sends data back to register page through onNext.
Output:
- Valid form data is saved in RegisterPage state.
- UI advances to map step.

2.3 LandMapSelector
File:
- components/register/LandMapSelector.tsx
Input:
- initialAddress from FarmerDetailsForm.
- Farmer search query.
- Farmer clicks on map points.
Source:
- Address from register form.
- Nominatim search API for geocoding.
- Leaflet map click events.
- ESRI satellite and label tile layers.
Processing:
- If address is provided, it tries to geocode address + Tamil Nadu India.
- Pans/zooms map to search result.
- Farmer clicks points to create land boundary.
- Double click or Done completes polygon.
- Requires at least 3 points before moving forward.
- onChange returns coordinates as [{lat,lng}].
Output:
- landCoordinates array is saved in RegisterPage state.
- Farmer sees drawn land polygon on satellite map.

2.4 Password Step
Input:
- password
- confirmPassword
Source:
- Farmer typing.
Processing:
- Client checks password match.
- Client checks real-app password strength:
  - minimum 8 characters
  - uppercase letter
  - lowercase letter
  - number
  - special character
- Show/hide toggles input type.
- Enter submits/advances when fields are valid.
Output:
- Valid password is included in registration payload.

2.5 Create Account Button
Input:
- formData from FarmerDetailsForm
- coords from LandMapSelector
- password
- verified phone OTP
Source:
- Register page state.
Processing:
- Calls POST /api/auth/register.
- API validates body.
- API verifies OTP for the normalized phone number.
- API checks duplicate farmer_id and phone.
- Password is hashed.
- Profile is stored in DynamoDB farmer_profiles.
Output:
- Success: redirects to /login?registered=1.
- Failure: error shown.

4.3 TodayActionBanner
File:
- components/dashboard/TodayActionBanner.tsx
Input:
- current weather
- forecast array
- all active crop plans
Source:
